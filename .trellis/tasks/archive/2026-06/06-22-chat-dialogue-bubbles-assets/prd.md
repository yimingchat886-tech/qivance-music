# Fix Chat Dialogue Bubbles And Shared Assets

## Goal

Fix `chat_dialogue_mv` so `run_4c182290a78145c5` and later production runs render actual dialogue bubbles in the captured `visual.mp4`/`final.mp4`, and redesign chat UI assets so generated projects reference reusable standard assets instead of storing copied avatar/icon files.

## What I Already Know

* User reported two issues for `run_4c182290a78145c5`: dialogue bubbles do not appear, and generated avatar/icon files are copied into the project asset folder instead of being reusable symlinks.
* `run_4c182290a78145c5` belongs to `projects/3test`, with artifacts under `projects/3test/data/chains/chat_dialogue_mv/**` and `projects/3test/exports/chat_dialogue_mv/**`.
* `projects/3test/data/chains/chat_dialogue_mv/runtime_timeline.json` contains 32 message events; `msg_001` starts at `9.661s`.
* `projects/3test/exports/chat_dialogue_mv/visual.mp4` at `12s` and `60s` shows only the chat shell and no message bubbles, while `qa_report.json` says `status=passed`.
* The runtime HTML itself can show bubbles: a CDP real-time probe calling `window.__qivanceChatRuntime.play()` and waiting `12.5s` showed `msg_001` visible with class `row right entered read-on`.
* Current generated runtime assets under `projects/3test/video/html-video/.html-video/projects/3test/assets/**` are regular files, not symlinks.
* Current source code copies packaged assets in `src/lib/chat-dialogue/chat-runtime-html.ts` from `src/lib/chat-dialogue/assets/default_avatars/**` and `src/lib/chat-dialogue/assets/status_bar_icons/**`.
* Current V4 spec says runtime HTML generation must copy packaged local assets; this task intentionally changes that contract to reusable standard-asset links.

## Assumptions

* The standard asset library can live in repo-owned chat-dialogue assets, while generated project folders should symlink to that library instead of copying bytes.
* WSL/Linux symlinks are acceptable for generated project runtime assets.
* Existing project-local `chat_ui_profile.json` overrides must keep working, but standard A/B/C profiles should become the default reusable path for avatar and name selection.
* No new dependency is needed.

## Requirements

* Browser recording must capture the runtime after playback actually starts, so message bubbles appear in `visual.mp4` when timeline events occur.
* Browser recording must not mark a run passed when the captured visual contains no visible message bubbles after the first message time.
* Generated runtime and static-fallback chat assets for packaged status icons and standard avatars must be symlinks to the standard asset library, not copied files.
* Standard chat UI profiles must define A/B/C reusable avatar/name presets. The default profile remains deterministic, and explicit project `chat_ui_profile.json` values may still override names/status/avatar paths.
* Keep artifact paths stable: runtime HTML remains under `video/html-video/.html-video/projects/<project_id>/runtime/chat_dialogue_mv.html`, visual under `exports/chat_dialogue_mv/visual.mp4`, final under `exports/chat_dialogue_mv/final.mp4`.
* Keep source changes scoped to `src/lib/chat-dialogue/**`, directly relevant tests, and the backend spec update required by the changed asset contract.

## Technical Approach

* First fix the recorder path, not the runtime HTML playback logic: real-time CDP playback proves the runtime can show bubbles, while the generated video stays empty.
* Use the smallest recorder change that makes playback state observable before capture and adds a guard against all-empty message output.
* Replace the current runtime/static `copyFile` asset staging with relative symlink staging for packaged icons and standard avatars.
* Keep the A/B/C profile table small and static. Do not add an asset registry service, database model, dependency, or broad media library.

## Decision (ADR-lite)

Context: The prior task fixed a render hang by adding CDP timeouts and asset copying. The new failure is subtler: rendering completes and QA passes, but the browser capture never shows message rows. Separately, copied runtime assets make generated projects carry duplicated avatar/icon files.

Decision: Treat this as a follow-up contract correction: make browser recording prove visible message output, and make generated chat runtime assets symlink to standard packaged assets with A/B/C identity presets.

Consequences: This keeps the existing runtime artifact layout and avoids a new asset service. It depends on symlink-capable local project storage; if non-WSL deployment later needs copy fallback, that should be a separate explicit portability task.

## Out Of Scope

* Rebuilding the chat UI design.
* Changing scheduler task order, manifest schema, DB schema, or export paths.
* Adding a new dependency or generic asset registry.
* Re-running every existing project unless requested after focused verification passes.
